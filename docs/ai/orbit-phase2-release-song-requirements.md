# Orbit Phase 2: 楽曲DB再設計（リリース中心）要件

- Issue: https://github.com/kikun-dev/personal-hub/issues/58
- 更新日: 2026-03-09
- 追補日: 2026-07-02（Issue #129 / #141 / #177 / #207、ADR 0007）

## 背景

既存の楽曲機能（`orbit_songs` 中心）は、今回の要件で必要な「リリース中心管理」「複数リリース再収録」「人物マスタ参照クレジット」「MV/衣装/特典映像」を十分に表現できない。
そのため、Phase 2 の楽曲領域はデータモデルとアプリ層を再設計する。

## 決定事項（仕様確定）

- 楽曲とリリースは M:N とし、中間テーブルでリリースごとの曲順を保持する。
- グループはリリースと楽曲の双方で保持する。楽曲側のグループはフォーメーション割当と一覧表示の基準に使う。
- 人物マスタを新設し、作詞/作曲/編曲/振付/MV監督/衣装担当は人物参照で管理する。
- リリースタイプは `single` / `album` / `best` / `compilation` / `digital_single` / `other`。
- ナンバリングは `single` / `album` のみ必須（整数）。`best` / `compilation` / `digital_single` / `other` では未設定必須。
- 楽曲時間は現行モデルでは保存しない（`duration_seconds` は Issue #141 で廃止）。
- 楽曲には単一ラベル（表題/選抜/アンダー/全員/ソロ/ユニット/期別、任意）を保持する。
- 選抜ポジションは楽曲ラベルとフォーメーションから導出し、リリース×メンバーで手動保持する情報は福神/休業中 overlay に限定する（ADR 0007）。
- MVは「存在する場合リンク必須」。監督/ロケ地/配信日/メモは任意。
- MVとは別に、楽曲関連動画（Dance Practice/ひなリハ、コール動画）を保持する。監督は持たず、URL必須・配信日/メモ任意とする。
- MV/関連動画に配信日がある場合、トップの月カレンダー/選択日/今日はなんの日へ動画配信イベントとして合成表示する。
- アートワーク・特典映像・MV・衣装は未設定可（空配列/NULL可）。
- 画像（リリースアートワーク、衣装画像）は Supabase Storage object path で保存する。
- 既存 `orbit_songs` 系は移行せず、置換（ゼロ再構築）する。

## 必須/任意

### リリース

- 必須: `title`, `group`, `type`
- 任意: `numbering`（ただし `single/album` では必須）, `releaseDate`, `artworkPath`, `bonusVideos`, `participants`

### 楽曲

- 必須: `title`, `release紐付け`
- 任意: `label`, `generation`, `credits`, `formation`, `mv`, `videos`, `costumes`

## 機能要件

### 1. リリース管理

- リリースごとにタイトル、グループ、タイプ、ナンバリング、リリース日を管理できる。
- 版種ごとの特典映像を複数行で管理できる（例: Type-A, Type-B）。
- リリース参加メンバーを複数選択できる。
- リリース曲目アートワークを保持できる。

### 2. 楽曲管理

- 楽曲は1件以上のリリースに紐づく。
- リリースごとの曲順（トラック順）を保持できる。
- 楽曲ごとのグループとラベルを保持できる。
- 作詞/作曲/編曲/振付を役割ごとに複数人登録できる。
- MVと関連動画（Dance Practice/ひなリハ、コール動画）を登録できる。
- 衣装を複数登録でき、衣装ごとに担当者と画像を保持できる。

### 3. フォーメーション

- 列数と列ごとの人数を設定できる。
- 参加メンバーは既存 `orbit_members` から選択し、かつ当該リリース参加者から選択する。
- 列ごとの人数設定と割当メンバー数は一致必須。
- 選抜/アンダー/期別、列、センターはフォーメーションと楽曲ラベルから導出する。
- 福神/休業中はフォーメーションだけでは判断できないため、リリース×メンバーの overlay として保持する。
- 櫻坂46 1st〜5th Single の櫻エイト期は、`label = title` の代表トラックを基準に特別ルールで導出する。

### 4. MV

- MVあり/なしを管理できる。
- MVありの場合はリンク必須。
- 監督、ロケ地、配信日、メモは任意。

### 5. 関連動画

- 関連動画の種別は `dance_practice` / `call`。
- `dance_practice` は櫻坂46で「Dance Practice」、日向坂46で「ひなリハ」と表示する。ラベルが解決できないグループでは表示対象外にする。
- `call` は「コール動画」として表示する。
- 関連動画ありの場合はリンク必須。配信日、メモは任意。
- 収録曲には MV / Dance Practice・ひなリハ / コール動画の有無をバッジ表示する。
- 配信日がある MV/関連動画は、トップの月カレンダー/選択日/今日はなんの日に「曲名（種別）」で表示し、リンクは動画URLへ外部リンクとして開く。

## データモデル方針（論理）

- `releases`: リリース本体
- `tracks`: 楽曲本体
- `release_tracks`: リリースと楽曲の中間（曲順を保持）
- `people`: 人物マスタ（表示名）
- `track_credits`: 楽曲クレジット（role + person）
- `release_members`: リリース参加メンバー
- `release_member_positions`: リリース×メンバーの選抜ポジション overlay（福神/休業中）
- `track_formations` / `track_formation_members`: フォーメーション列定義と割当
- `track_mvs`: MV情報
- `track_videos`: 楽曲関連動画
- `release_bonus_videos`: 特典映像
- `track_costumes`: 衣装情報

注: 物理テーブル名は既存 `orbit_` 命名規則に合わせて実装時に確定する。

## 画面スコープ

- Admin: リリース/楽曲の一覧・作成・編集・削除（関連子要素含む）
- Public: 一覧/詳細（リリース情報、楽曲詳細、フォーメーション表示）

## 非対象（Non-goals）

- 既存楽曲データの移行
- ライブ/セットリスト実装
- 匿名公開向けRLS方針変更

- 新モデルで公開一覧/詳細が成立する。
