# ADR 0001: 基本アーキテクチャとAI分担方針

## Status
Accepted

## Context（背景）

本プロジェクトは個人開発を前提とし、長期的に複数アプリを共通基盤の上で育てていく。

- 推し活情報集約
- 推し活に紐づく家計管理
- タスク管理
- 将来的な拡張

設計の一貫性と履歴の可視化が重要である。

また、開発にはAIを活用する：

- Claude：設計提案 + 実装
- Codex：レビュー + リファクタリング

AI活用前提で、設計がブレない構造を採用する必要がある。

---

## Decision（決定）

以下を採用する：

### 1. 技術前提（A運用）

- Next.js + TypeScript
- 層分離アーキテクチャ

### 2. 層構造

UI → UseCase → Data/Repository

- UI：表示とユーザー操作
- UseCase：ビジネスロジック
- Data：外部I/O・副作用

依存方向の逆転は禁止。

### 3. 設計ログの集約方針

- 設計議論は Issue に残す
- 長期影響のある判断は `docs/decisions/` にADRとして保存
- PRは実装ログとする

### 4. AI分担

- Claude：設計提案・意思決定明示・実装
- Codex：レビュー・構造整理・リファクタ

設計変更はIssueで決定し、PRで議論しない。

---

## Consequences（結果・影響）

### 良い点
- 長期的に構造が崩れにくい
- 設計履歴が追跡可能
- AIが迷いにくい
- 依存方向が明確

### 悪い点
- 初期の記録コストが増える
- 小規模変更でもDecisionを残す判断が必要

---

## Alternatives Considered（検討した代替案）

### 1. 単層構造（UIにロジックを含める）
- 実装は速い
- 長期で肥大化しやすい
- テスト困難

→ 採用しない

### 2. 技術選定もAIに完全委任（B運用）
- 柔軟性は高い
- 設計が揺れやすい

→ 今回はA運用（技術前提固定）を採用

---

## Review Trigger（見直し条件）

以下が発生した場合、このADRを見直す：

- アプリ数が大幅に増加
- モバイル専用アプリを追加
- サーバーサイド構成を変更
- 共通基盤が肥大化した場合

---

## Notes

このADRはプロジェクトの「憲法レベル」の決定である。
変更する場合は新しいADRを作成し、置き換える。
