# ADR 0007: Orbit 選抜ポジションのフォーメーション一元管理

## Status
Accepted

## Context（背景）

Orbit では楽曲ごとのフォーメーションと、シングルごとの選抜ポジションを別々に保持していた。

- フォーメーション: 楽曲単位。列、メンバー、センターを保持する。
- 選抜ポジション: リリース×メンバー単位。選抜/アンダー/期別、列、センター、福神/櫻エイト、休業中を保持する。

この構成では、同じ「誰がどの枠か」という情報が複数箇所に存在し、楽曲フォーメーションとメンバー詳細の選抜ポジション表示がずれる可能性がある。

一方で、選抜/アンダー/期別、列、センターは楽曲フォーメーションから導出できる。手動で保持する必要があるのは、フォーメーションだけでは判断できない overlay 情報に限られる。

## Decision（決定）

選抜ポジションは、楽曲フォーメーションを単一ソースとして導出する。

### 1. フォーメーションから導出する情報

以下は DB に保存せず、楽曲フォーメーションと楽曲ラベルから導出する。

- 選抜
- アンダー / BACKS / ひなた坂
- 期別
- 列
- センター

楽曲ラベルの優先順は、表題、選抜、アンダー、期別の順とする。
同一ラベルの曲が複数ある場合は、リリース内の最小 `track_number` を代表トラックとして扱う。
期別曲は `generation` ごとに別代表として扱い、別期の期別曲を同一視しない。

### 2. 手動保持する overlay

リリース×メンバーで手動保持する情報は以下に限定する。

- 福神
- 休業中

櫻エイトは手動 overlay として保持しない。櫻坂46の初期シングル特有の導出ルールとして扱う。

### 3. 櫻エイト期の特別ルール

櫻坂46 1st〜5th Single は櫻エイト期として扱う。

櫻エイト期では `label = title` の代表トラックだけを表題曲として評価する。

- 表題曲の1・2列目: 櫻エイト
- 表題曲の3列目以降: 接頭辞なしの `N列目`
- 表題曲に居ない参加メンバー: 列なし `BACKS`

`label = senbatsu` の選抜カップリングや BACKS 曲は、櫻エイト期では表題側の列導出に使わない。

### 4. 保存モデル

`orbit_release_member_positions` は導出済みポジションを保持せず、overlay 保存用の薄いテーブルとして扱う。
リリースフォームでは、選抜/アンダー/列/センターを手動入力させず、福神と休業中のみを指定する。

## Alternatives Considered（検討した代替案）

### 選抜ポジションを従来どおり保持し、生成ボタンで補助する

フォーメーションから選抜ポジションを生成し、保存時に整合チェックする案。
既存UIを活かしやすいが、二重管理自体は残るため、ズレを構造的には防げない。

### フォーメーションと選抜ポジションを両方保持し、警告だけ出す

実装コストは小さいが、最終的な整合性が利用者の手作業に依存する。
今回の「一元管理」要望とは合わない。

## Consequences（結果・影響）

### 良い点

- フォーメーションとメンバー詳細の選抜ポジション表示が構造的にずれにくくなる。
- リリースフォームでの二重入力が減る。
- センターや列の変更は楽曲フォーメーション側に集約できる。
- 櫻坂46初期の櫻エイト期も、既存の `SelectionTier` を増やさずに表現できる。

### 悪い点

- メンバー詳細の選抜ポジション表示は、対象シングルに楽曲ラベルとフォーメーションが登録されていることに依存する。
- 複数表題曲や期別曲など、代表トラックの決定ルールを理解してデータ登録する必要がある。
- 櫻エイト期のようなグループ固有ルールは、導出ロジック側に明示的な特別扱いを持つ。
