# ADR 0007: Orbit 選抜ポジションと参加メンバー・フォーメーションの正典

## Status
Accepted

## Context（背景）

Orbit では、楽曲フォーメーションとリリース×メンバーの選抜ポジションを別々に保持していた。

- フォーメーション: 楽曲単位。列、メンバー、センターを保持する
- 選抜ポジション: リリース×メンバー単位。選抜/アンダー/期別、列、センター、福神/櫻エイト、休業中を保持する

同じ「誰がどの枠か」という情報が複数箇所に存在すると、楽曲フォーメーションとメンバー詳細の表示がずれる。そのため Issue #177 では、選抜/アンダー/期別、列、センターを楽曲フォーメーションから導出し、手動保持をフォーメーションだけでは判断できない overlay に限定した。

その後、Issue #425 で次の課題が明確になった。

- 楽曲単位の参加メンバーを、フォーメーションと独立して保持できない
- 「参加メンバーやセンターは判明しているが、フォーメーションは未解禁」という状態を表現できない
- 楽曲参加メンバーの候補に使うリリース参加メンバーと、楽曲固有の参加メンバーの責務が曖昧である
- ライブでは披露メンバーとフォーメーションを別管理しているため、楽曲マスタから何を初期値として渡すかを統一する必要がある

一元管理の目的は維持しつつ、参加者集合、センター、配置、リリース参加者をそれぞれ適切な正典へ分ける。

## Decision（決定）

### 1. 楽曲参加メンバーとセンター

`orbit_track_members` を楽曲参加メンバーの正典として追加し、センターを `is_center` で保持する。

- 参加メンバーは楽曲単位で保持する
- センターは最大2人とし、必ず楽曲参加メンバー内に限定する
- 参加メンバーとセンターは、フォーメーションがなくても登録できる
- `orbit_track_members` は列、配置順、ポジションを保持しない
- センターの正典を `orbit_track_members.is_center` へ移し、`orbit_track_formation_members.is_center` は移行後に廃止する

これにより、センターをフォーメーションから導出する従来の決定を改訂する。

### 2. 楽曲フォーメーション

フォーメーションは任意とし、列と列内の並び順の正典として扱う。

- フォーメーションが存在する場合、全列のメンバー集合と `orbit_track_members` を完全一致させる
- 部分集合のフォーメーションは許可しない
- センターは、フォーメーションが存在する場合のみ1列目に含まれることを必須とする
- 入力途中の未配置状態はUIで表現できるが、フォーメーションを保存する時点では全参加メンバーの配置を必須とする

### 3. 選抜ポジションの導出

以下はDBへ重複保存せず、代表トラックの楽曲ラベルとフォーメーションから導出する。

- 選抜
- アンダー / BACKS / ひなた坂
- 期別
- 列

センターだけはフォーメーションから導出せず、代表トラックの `orbit_track_members.is_center` を参照する。フォーメーション未登録でも、センター楽曲として表示できる。

楽曲ラベルの優先順は、表題、選抜、アンダー、期別の順とする。同一ラベルの曲が複数ある場合は、リリース内の最小 `track_number` を代表トラックとして扱う。期別曲は `generation` ごとに別代表として扱い、別期の期別曲を同一視しない。

### 4. リリース参加メンバーと手動 overlay

`orbit_release_members` はリリース全体の参加者・休業情報として独立管理し、楽曲参加メンバーの和集合から自動導出しない。

- アルバムや再収録では、音源参加者とリリース時点のメンバーが一致しない場合がある
- 楽曲参加メンバーがリリース参加メンバー外でもDBでは拒否しない
- 不一致は管理UIの警告として扱い、登録を妨げるハード制約にはしない

`orbit_release_member_positions` は導出済みポジションを保持せず、次の手動 overlay だけを保存する。

- 福神
- 休業中

櫻エイトは手動 overlay として保持せず、櫻坂46の初期シングル固有の導出ルールとして扱う。

### 5. 櫻エイト期の特別ルール

櫻坂46 1st〜5th Single は櫻エイト期として扱い、`label = title` の代表トラックだけを表題曲として評価する。

- 表題曲の1・2列目: 櫻エイト
- 表題曲の3列目以降: 接頭辞なしの `N列目`
- 表題曲にいない参加メンバー: 列なし `BACKS`

`label = senbatsu` の選抜カップリングやBACKS曲は、櫻エイト期では表題側の列導出に使わない。

### 6. ライブの披露メンバーとフォーメーション

ライブ側も、楽曲と同じ「参加者＋センター」「任意の配置」というモデルに揃える。

- 披露メンバーとセンターの正典は `orbit_setlist_item_members`
- センターは披露メンバー内の最大2人とし、フォーメーションがなくても登録できる
- ライブフォーメーションは任意
- フォーメーションが存在する場合、全列のメンバー集合と披露メンバーを完全一致させる
- センターは、フォーメーションが存在する場合のみ1列目に含まれることを必須とする
- メンバーを外す場合は、配置とセンターからも同時に外す
- メンバーを追加した場合は未配置として扱い、フォーメーションがある状態では全員を配置するまで保存しない

### 7. 楽曲マスタからライブへの初期値連携

「オリメン」は楽曲参加メンバーを意味する。フォーメーションの有無をオリメン判定には使わない。

- 公演の出演可能メンバー = ライブ基準ロスター − 当該公演の休演メンバー
- 反映対象 = 楽曲参加メンバー ∩ 公演の出演可能メンバー
- 楽曲フォーメーションがある場合は、参加メンバー、センター、フォーメーションを同じ判定結果から一単位として反映する
- フォーメーションがなくても、参加メンバーとセンターは反映できる
- 公演日時点の卒業と当該公演の休演は自動除外し、件数と理由を情報表示する
- 現役だがライブ基準ロスター外の場合は自動追加せず、ライブ編集への登録導線を表示する
- 楽曲参加メンバーが未登録の場合は、楽曲編集への登録導線を表示する
- コピー後のライブデータは公演固有の実績として独立し、楽曲マスタの変更へ自動追従しない
- 代打、センター変更、配置変更はライブ側で手動調整する

### 8. 既存データ移行

- フォーメーションがある楽曲は、全列のdistinctメンバーを `orbit_track_members` へbackfillする
- `orbit_track_formation_members.is_center` を `orbit_track_members.is_center` へbackfillする
- フォーメーションがない楽曲は空のままとし、リリース参加メンバーから推測補完しない
- メンバー詳細の参加楽曲とセンター楽曲は、移行後に `orbit_track_members` を正典とする

## Alternatives Considered（検討した代替案）

### フォーメーションを楽曲参加メンバーとセンターの正典として維持する

変更は小さいが、参加メンバーやセンターだけ判明している状態を表現できず、フォーメーション未解禁時の情報を失う。

### フォーメーションに公開状態・部分登録を持たせる

draft / member-only / published などを同じ集約で管理する案。公開状態まで表現できる一方、現在必要な「集合と配置の分離」に対して複雑であり、履歴・版管理の要件が固まる前には過剰と判断した。

### フォーメーションと参加メンバーを独立保存し、不一致は警告だけにする

入力の自由度は高いが、恒常的な不整合を許し、「未配置」と「入力漏れ」を区別できない。フォーメーションが存在する場合は完全一致を不変条件とする。

## Consequences（結果・影響）

### 良い点

- フォーメーション未解禁でも、楽曲参加メンバーとセンターを記録できる
- 楽曲とライブで同じ概念モデルを利用できる
- 参加者集合とフォーメーションの完全一致により、二重管理のずれを防げる
- リリース参加者を独立した事実として維持できる
- コピー時の想定内の除外と、登録漏れへの行動可能な導線を分けられる

### 悪い点

- migration、型、Repository、UseCase、管理UI、公開表示、ライブ連携を段階的に移行する必要がある
- 入力途中には未配置状態が生じるため、UIと保存境界の両方で整合を扱う必要がある
- 楽曲参加メンバー、リリース参加メンバー、ライブ出演メンバーという異なる集合の責務を理解して登録する必要がある
- 複数表題曲や期別曲など、代表トラックの決定ルールは引き続き必要である
