# ADR 0011: Wiki的静的ページ集のコンテンツ管理と Markdown レンダリング

## Status

Accepted

## Context（背景）

オーディション関係・リリーススケジュール・遠征持ち物リスト・外部リンク集など、
「参照」がメインの静的ページ集（Wiki的なもの）を Orbit（oshikatsu-web）に追加する（Issue #312）。
一覧からページを開き、目次から見出しへジャンプできる読み物ページで、
楽曲・ライブのような動的データはここには置かない。

実装前に2つの判断が必要だった。

### 1. コンテンツをどこに置き、どう編集するか

- 案A: リポジトリ内 Markdown ファイル + ビルド時読み込み（編集 = ソース編集 + デプロイ）
- 案B: DB テーブル + 管理画面から作成・編集
- 案C: A で開始し、必要になったら B へ移行（閲覧側をコンテンツ源非依存に組む）

案A/C は実装コスト最小で Git 履歴も残るが、編集のたびにコミット + デプロイが必要で、
スマホから直せない。案B は編集UIの実装が要るが、既存の admin ロール・RLS・
shared cache（ADR 0006/0008）のパターンにそのまま乗り、デプロイ不要で出先からも編集できる。

### 2. Markdown をどうレンダリングするか

- 案R1: react-markdown（remark ベース）+ remark-gfm を導入
- 案R2: 自前パーサ（`lib/linkParser.ts` の延長）

R2 は依存ゼロだが、見出し・リスト・表・リンクを満たす時点で事実上の
Markdown エンジン自作になり、実装・保守コストが新規ライブラリ1つの導入コストを大きく上回る。

## Decision（決定）

### 1. コンテンツは DB（`orbit_wiki_pages`）に置き、管理画面から編集する（案B）

- 出先のスマホからも編集したい運用を優先し、最初から DB + 管理画面編集を採用する
- テーブルは slug / title / body_markdown / sort_order を持つグローバルデータとし、
  RLS は既存方針（SELECT = admin/viewer、書き込み = admin。ADR 0008）をそのまま適用する
- 閲覧は shared read cache（ADR 0006）に載せ、専用のキャッシュタグで失効させる
- 版管理・下書きは持たない（個人運用。必要になったら別途検討）

### 2. レンダリングは react-markdown + remark-gfm を採用する（案R1）

- React コンポーネントとして描画するため `dangerouslySetInnerHTML` 不要
  （コードベースの「dangerouslySetInnerHTML ゼロ」を維持する）
- 見出しへの id 付与と目次生成は、TOC 抽出と描画で同じスラッグ関数を共有する
  小さな自前実装とし、rehype プラグインの追加は最小限に留める

## Consequences（結果・影響）

### 良い点

- スマホ・PC どちらからでもデプロイなしにページを追加・編集できる
- 認可・キャッシュ・管理画面 CRUD が既存パターンの再利用で済み、設計判断が増えない
- react-markdown により HTML 直挿入を避けたまま、表・リストを含む Markdown が描ける

### 悪い点

- 「新規ライブラリ最小限」方針の例外として react-markdown（+ remark-gfm）への依存が増える
  （remark エコシステムのメジャーアップデート追随コストを引き受ける）
- コンテンツが Git 履歴の外に出る（ページの変更履歴・レビューは残らない）
- 編集UI（フォーム・プレビュー）の実装コストが発生する（Issue #314）
