# 現状調査（リファクタリング監査）

対象アプリを観点別に調査し、`docs/advisor/` に監査ドキュメントを残す。
`docs/advisor/006-oshikatsu-web-current-state-audit.md` が見本。

## 手順

### 1. 前回監査の確認

- `docs/advisor/` の同一対象の過去監査を読み、前回の指摘・Issue 化状況を把握する
- 2回目以降の監査では、指摘ごとに **改善済み / 残存 / 新規** を区別して報告する
  （前回と同じ内容を新規発見のように書かない）

### 2. 観点別調査（優先順位順）

1. **セキュリティ（認証・認可含む）**: middleware/proxy のガード範囲、RLS ポリシー、
   Server Actions の認証チェック、service role の利用箇所、XSS ベクター
   （`dangerouslySetInnerHTML`、URL スキーム検証）、secrets の扱い
2. **データアクセス**: repository パターンの一貫性、エラーハンドリング（`RepositoryError` / `Result`）、
   N+1、型安全性（キャスト・手書き Row 型の量）、トランザクション境界
3. **アーキテクチャ・共通化**: 層の依存方向違反（`rules/architecture.md`）、
   巨大ファイル（`wc -l` 上位）、アプリ間・ファイル間の重複、god module 化の兆候
4. **UI設計・状態管理**: client/server コンポーネント境界、フォームの状態管理パターンの重複、
   エラーバウンダリ・404 の整備状況
5. **パフォーマンス**: キャッシュ戦略（`unstable_cache` / タグ失効の粒度）、
   全件取得のスケール限界、`next/image`、バンドルへの影響
6. **横断**: 可観測性（ログ・監視）、依存パッケージ健全性、typecheck / lint の通過

### 3. 調査の規律

- 指摘には必ず根拠を付ける: `ファイルパス:行` / migration 番号 / 実行したコマンドの結果
- ADR・Issue で**意図的に決定済み**の事項は、欠陥ではなく「決定とその前提」として扱う
  （例: service role read cache は ADR 0006 の決定。前提が崩れていないかを見る）
- コードから確認できないこと（Supabase ダッシュボード設定等）は「要確認」として明示し、
  勝手に問題と断定しない
- 良い点も記録する（次回監査の基準線になる）

### 4. 監査ドキュメントの作成

- `docs/advisor/NNN-<対象>-current-state-audit.md`（連番は advisor 内の最大 + 1）
- 構成は 006 に合わせる: 作成日・対象・調査観点 → 総評 → 観点別（良い点 / 指摘事項）→
  **優先順位付きリファクタリング計画表**（P1/P2/P3、規模・効果付き）→ Issue 化状況
- 日本語ベースで記載する

### 5. Issue 化とドキュメント反映

- 改善項目の Issue 化を提案し、承認されたら `/design-issue` の手順で起票する
- 新規の技術的負債は `docs/orbit-roadmap.md` の「技術的負債・既知の制限」表に追加する
- 前回監査の指摘が解消済みなら、該当ドキュメント・roadmap の表を更新する（`/doc-sync` 相当）

## 完了条件

- 監査ドキュメントが根拠付き・優先順位付きで作成されている
- 前回監査との差分（改善済み / 残存 / 新規）が明示されている（2回目以降）
- Issue 化・roadmap 反映が完了しているか、提案として提示されている
